# 인스타그램 포스팅 준비

이 스킬은 최근 작업 내용을 요약해 인스타그램에 올릴 **사진(스크린샷)**, **릴스용 무음 영상
(자막 포함) + 녹음 대본**, **캡션 텍스트(해시태그 포함)**를 준비한다. Instagram API 연동은
하지 않고, 음성 합성(TTS)도 쓰지 않는다 — 나레이션은 사용자가 직접 녹음하는 걸 전제로 한다
(2026-07-12에 사용자가 "AI 티 안 나는 음성"을 요구했는데, 이 환경에서 그걸 만족하는 유일한
방법은 실제 사람이 녹음하는 것뿐이라고 판단해서 이렇게 결정함 — Windows 기본 TTS는 로봇 음성
티가 확실히 나고, ElevenLabs 등 고급 TTS는 별도 계정·API 키·비용이 필요해서 사용자 승인 없이
진행할 수 없음). 사용자가 결과물을 보고 직접 앱에서 올리는 것을 전제로 한다.

## 1. 무엇이 새로 추가됐는지 파악 (조사 → 반드시 사용자에게 확인)

**스크린샷/영상을 만들기 전에 항상 사용자에게 먼저 물어본다.** 혼자 짐작해서 밀어붙이지
말 것 — 이게 2026-07-12에 사용자가 명시적으로 요청한 규칙이다.

1. `TODO.md`의 "최근 완료" 섹션에서 가장 최근 날짜 블록을 읽는다. 이 프로젝트는 작업할 때마다
   이 섹션에 완료 항목을 기록해두므로, 여기서 "이번에 뭘 했는지" 후보 목록을 뽑는다.
2. `instagram_posts/.last_post_commit` 파일이 있으면 그 안의 커밋 해시 이후 `git log --oneline`을
   같이 참고해서, 이미 포스팅했던 내용과 겹치지 않게 후보를 추린다. 파일이 없으면(첫 실행)
   TODO.md의 최신 완료 블록 전체를 후보로 삼는다.
3. 후보를 정리했으면 **AskUserQuestion으로 최소 아래 항목을 확인한다** (이미 명확히 말해준
   내용이 있으면 그 항목은 스킵해도 되지만, 기본은 물어보는 것):
   - **어떤 내용을 다룰지**: 후보로 뽑은 완료 항목 중 무엇을 이번 포스팅 주제로 삼을지
     (여러 개면 옵션으로 나열해서 고르게 하거나, 하나로 묶을지 확인)
   - **사진만 vs 영상(릴스)도**: 정적 이미지만 준비할지, 3번 항목의 릴스까지 만들지
   - **분량**: 사진이면 몇 장 정도, 영상이면 대략 몇 초 정도로 할지 (기본값 제시하고 확인받는
     정도면 충분 — 예: "사진 6~8장, 영상은 20초 안팎으로 할까요?")
   - 특별히 강조하고 싶은 포인트나 톤이 있는지 (없으면 기존 톤 가이드 그대로 진행)
4. 답변을 받은 뒤에 2번(스크린샷)·3번(영상) 단계로 넘어간다. 사용자가 답변한 범위를 벗어나는
   내용을 임의로 추가하지 않는다.

## 2. 스크린샷 준비 (모바일 세로 비율, 카드/섹션 단위로 분할)

**한 장짜리 풀페이지 스크린샷을 그대로 쓰지 말 것.** 인스타 피드는 세로 비율이 4:5 정도까지만
잘 보이는데, 페이지 전체를 한 장으로 찍으면 훨씬 더 길어서 피드에서 위아래가 잘려 보인다.
그렇다고 고정 픽셀 간격(예: 600px마다)으로 기계적으로 잘라도 안 된다 — 카드나 문장 중간이
잘려서 이미지 한 장만 보면 무슨 내용인지 이해가 안 되는 문제가 있었다 (2026-07-12에 실제로
겪음). **반드시 카드/섹션 경계에서만 잘라야 한다.**

### 2-1. 전체 페이지 스크린샷 먼저 확보

- 배포된 사이트(`https://littlecub125.github.io/newsletter_industry_update/`)를 대상으로,
  이번 작업과 관련 있는 페이지 위주로 스크린샷을 찍는다. 배포가 최신 상태인지 확실하지 않으면
  먼저 `git log`로 마지막 커밋과 `gh-pages` 배포 시점을 확인하거나, 로컬 서버(`python -m
  http.server`, `web/` 폴더)를 대신 써도 된다.
- PowerShell + Edge 헤드리스로 캡처한다. **`--window-size`의 너비는 480을 쓸 것** — 430처럼
  실제 모바일 뷰포트 값을 그대로 넣으면 헤드리스 캡처 특성상 우측이 잘리는 문제가 있었다.
  480이면 잘림 없이 깔끔하게 나온다.
  ```powershell
  $edge = "C:\Program Files (x86)\Microsoft\Edge\Application\msedge.exe"
  & $edge --headless --disable-gpu --hide-scrollbars --force-device-scale-factor=1 `
    --window-size=480,<세로값> --virtual-time-budget=8000 --screenshot="<출력경로>.png" "<URL>"
  ```
  - `<세로값>`은 페이지 실제 콘텐츠보다 넉넉하게 크게 잡는다 (예: 2400~2800). 모바일에서는
    카드/그리드가 세로로 쌓여서 데스크톱보다 훨씬 길어진다 — 너무 작게 잡으면 뒷부분(하단 카드,
    푸터)이 통째로 안 찍힌다. 일단 크게 찍고, 아래쪽 여백이 많으면 자를 때 버리면 된다.
  - `--virtual-time-budget=8000`은 `fetch("news.json")` 같은 비동기 로딩이 끝날 시간을 준다.
    이걸 빼먹으면 뉴스 목록이 빈 채로 찍힌다.
  - 풀페이지 스크린샷은 `instagram_posts/<날짜>/full_page/`에 저장한다 (최종 포스팅용이
    아니라 자르기 위한 원본이므로 별도 폴더로 구분).

### 2-2. 흰 여백을 스캔해서 카드/섹션 경계 찾기

고정 간격으로 자르지 말고, 이미지에서 "완전히 흰 줄(카드 사이 여백)"이 넓게 이어지는 지점을
찾아서 그 지점을 자르는 기준으로 삼는다. PowerShell + System.Drawing으로 스캔한다:

```powershell
Add-Type -AssemblyName System.Drawing
function Find-Gaps($path, $minGapHeight) {
  $img = New-Object System.Drawing.Bitmap $path
  $w = $img.Width; $h = $img.Height
  $isWhiteRow = New-Object bool[] $h
  for ($y = 0; $y -lt $h; $y++) {
    $white = $true
    for ($x = 0; $x -lt $w; $x += 8) {
      $p = $img.GetPixel($x, $y)
      if ($p.R -lt 250 -or $p.G -lt 250 -or $p.B -lt 250) { $white = $false; break }
    }
    $isWhiteRow[$y] = $white
  }
  $gaps = @(); $start = -1
  for ($y = 0; $y -lt $h; $y++) {
    if ($isWhiteRow[$y]) { if ($start -eq -1) { $start = $y } }
    else { if ($start -ne -1) { $len = $y-$start; if ($len -ge $minGapHeight) { $gaps += [PSCustomObject]@{Start=$start;End=$y;Mid=[int](($start+$y)/2);Len=$len} }; $start=-1 } }
  }
  $img.Dispose(); return $gaps
}
Find-Gaps "<풀페이지.png경로>" 40 | Format-Table
```

- `minGapHeight`(마지막 인자)를 40~45 정도로 시작한다. 이러면 섹션 사이의 큰 여백(60px+)은
  잡히지만, 문단 안 줄바꿈이나 리스트 항목 사이의 자잘한 간격(보통 20px 이하)은 노이즈로
  걸러진다.
- 이렇게 찾은 경계만으로 카드 개수만큼 안 나뉘면(예: 요금제 카드처럼 카드 사이 간격이
  20px밖에 안 되는 경우), 자를 지점 후보들의 `Mid` 값을 하나씩 살펴보고 실제로 카드/섹션이
  바뀌는 지점인지 판단한다. 애매하면 일단 잘라서 Read 도구로 직접 열어 확인하고, 텍스트가
  잘렸으면 경계를 30~50px 정도 옮겨서 다시 자른다 — **이 확인 단계를 생략하지 말 것.**

### 2-3. 자르기

```powershell
Add-Type -AssemblyName System.Drawing
$img = [System.Drawing.Image]::FromFile("<풀페이지.png>")
$w = $img.Width
$y0 = <시작Y>; $y1 = <끝Y>; $th = $y1 - $y0
$tile = New-Object System.Drawing.Bitmap $w, $th
$g = [System.Drawing.Graphics]::FromImage($tile)
$g.DrawImage($img, (New-Object System.Drawing.Rectangle 0,0,$w,$th), (New-Object System.Drawing.Rectangle 0,$y0,$w,$th), [System.Drawing.GraphicsUnit]::Pixel)
$g.Dispose()
$tile.Save("<출력경로>.png", [System.Drawing.Imaging.ImageFormat]::Png)
$tile.Dispose(); $img.Dispose()
```

- 조각 이미지는 `instagram_posts/<날짜>/` 최상위에 `01_`, `02_` 순번을 붙여 저장한다
  (풀페이지 원본은 `full_page/` 하위에 남겨두되 이건 포스팅용이 아니라 참고용).
- 완전히 빈(흰색뿐인) 조각이 나오면 삭제한다.
- 카드 1개가 한 장에 다 안 들어가면 억지로 줄이지 말고 조각을 늘린다 — 장수보다 "잘림 없이
  읽히는 것"이 우선이다.

## 3. 릴스/쇼츠용 영상 준비 (무음 영상 + 자막 + 녹음 대본)

**전제**: 나레이션 음성은 사용자가 직접 녹음한다. 이 스킬은 (a) 자막이 이미 박힌 무음 영상,
(b) 같은 내용의 SRT 자막 파일, (c) 타이밍이 표시된 녹음 대본까지만 만든다. 사용자가 CapCut/
인스타 자체 편집기 등에서 이 무음 영상을 불러와 "영상 보면서 목소리 녹음" 기능으로 대본을
읽으면 완성된다.

### 3-0. 도구 확인

- `ffmpeg`이 설치돼 있는지 확인. 없으면 `winget install --id Gyan.FFmpeg -e --accept-source-agreements --accept-package-agreements`로 설치한다.
- 설치 직후엔 현재 Bash 세션 PATH에 안 잡힐 수 있다 — 아래처럼 실제 설치 경로를 찾아서 절대
  경로로 호출한다:
  ```bash
  find /c/Users/*/AppData/Local/Microsoft/WinGet/Packages -iname "ffmpeg.exe" 2>&1
  ```
- 한글 자막을 그리려면 한글 지원 폰트가 필요하다. Windows 기본 `C:\Windows\Fonts\malgunbd.ttf`
  (맑은 고딕 Bold)를 쓴다 — 이 프로젝트 웹사이트 CSS의 폰트 스택과도 맞아서 톤이 일관된다.

### 3-1. 장면(scene) 구성

- 이번에 준비한 인스타 포스팅용 조각 이미지(위 2번 단계 결과물) 중 5~7장을 골라 순서를 정한다.
  전체를 다 쓸 필요 없다 — 릴스는 15~25초 정도가 적당하므로, 장면당 3~4초 잡아서 총 길이를
  맞춘다.
- 각 장면에 한두 줄짜리 자막 문구를 쓴다. `TODO.md`의 완료 내역을 훅(hook) → 배경/이유 →
  결과 → 다음 계획 순서로 짧게 각색한다 (캡션과 같은 톤 — 4번 항목 참고).
- `instagram_posts/<날짜>/reel/scenes/captions.txt`에 아래 형식으로 저장한다. 줄바꿈은
  `\n`(리터럴 두 글자)으로 표기한다:
  ```
  01|<이미지파일명>|<지속시간(초)>|<자막 1행>\n<자막 2행>
  02|<이미지파일명>|<지속시간(초)>|...
  ```

### 3-2. 무음 영상 만들기 (ffmpeg)

`instagram_posts/<날짜>/reel/build_reel.ps1`에 아래 로직으로 스크립트를 작성해 실행한다
(과거 실행분 예시가 있으면 그대로 복사해서 캡션/이미지만 바꿔도 됨):

- 캔버스는 릴스 표준 규격인 **1080x1920 (9:16)**, 배경은 흰색.
- 각 장면 이미지는 **`scale=950:1380:force_original_aspect_ratio=decrease`** 로 최대
  950x1380까지만 줄이고, **`pad=1080:1920:(1080-iw)/2:110:color=white`** 로 캔버스에
  올린다 — **세로는 중앙 정렬하지 말고 위에서 110px 고정 오프셋으로 붙일 것.** 세로 중앙
  정렬을 하면 이미지마다 스케일된 높이가 달라져서, 아래에 그릴 자막 박스와 카드/버튼이
  겹치는 문제가 실제로 있었다 (2026-07-12). 위 수치(최대높이 1380 + 상단오프셋 110)면
  이미지 하단이 최대 1490까지만 내려오고, 자막은 그 아래 1650부터 시작해서 절대 안 겹친다.
- 자막은 `drawtext` 필터로 굽는다. `text=` 대신 **`textfile=`** 를 쓸 것 — 캡션에 작은따옴표나
  콜론 같은 특수문자가 있으면 필터 문자열 자체가 깨진다. 캡션 텍스트는 UTF-8(BOM 없이)로
  별도 .txt 파일에 저장한 뒤 그 경로를 넘긴다.
- **자막에 이모지를 넣지 말 것.** `malgunbd.ttf`에 이모지 글리프가 없어서 빈 사각형(□)으로
  깨진다 (2026-07-12에 실제로 겪음). 느낌표(!) 등 일반 문장부호로 대체한다.
- **Windows 경로를 필터 문자열에 넣을 때 콜론(`:`) 이스케이프 필수.** `C:\Windows\Fonts\...`
  처럼 드라이브 문자 뒤 콜론이 ffmpeg 필터 문법의 옵션 구분자와 충돌해서 파싱 에러가 난다.
  백슬래시를 슬래시로 바꾸고 드라이브 콜론 앞에 `\`를 붙인다:
  ```powershell
  $esc = ($path -replace '\\', '/') -replace '^([A-Za-z]):', '$1\:'
  ```
  `fontfile=`, `textfile=` 둘 다 이렇게 이스케이프한 경로를 넘긴다.
- **PowerShell에서 ffmpeg 실행 시 `2>&1`로 파이프하지 말 것.** ffmpeg는 정상 동작해도 버전
  배너 등을 stderr로 출력하는데, PowerShell 5.1에서 네이티브 명령의 stderr를 `2>&1 | ...`로
  받으면 각 줄이 에러로 취급돼 스크립트가 멈춘다 (2026-07-12에 실제로 겪음). `-loglevel error`
  옵션으로 ffmpeg 자체 출력을 최소화하고, 리다이렉트 없이 그냥 실행한다.
- 장면별로 `scene_NN.mp4`를 만든 뒤, concat demuxer로 이어붙인다 (`-f concat -safe 0`).
- 완성되면 최소 2~3개 프레임을 `ffmpeg -vf "select='eq(n\,<프레임번호>)'"`로 뽑아 Read 도구로
  직접 열어 확인한다 — 자막이 카드/버튼과 안 겹치는지, 한글이 깨지지 않는지, 이모지가 없는지
  꼭 확인할 것. 문제 있으면 캡션이나 pad 오프셋을 조정해서 다시 만든다.
- 최종 무음 영상은 `instagram_posts/<날짜>/reel/reel_silent.mp4`로 저장한다.

### 3-3. SRT 자막 + 녹음 대본 작성

- `instagram_posts/<날짜>/reel/reel_subtitles.srt`: 각 장면의 시작~끝 시각(HH:MM:SS,mmm)과
  자막 문구를 표준 SRT 포맷으로 저장한다. 영상엔 이미 자막이 박혀있지만, 편집 앱에 따로
  임포트하고 싶을 때를 위한 참고용이다.
- `instagram_posts/<날짜>/reel/narration_script.txt`: 자막과 같은 타이밍으로, 사람이 자연스럽게
  소리 내어 읽을 수 있는 구어체 대본을 쓴다. `[0:00 ~ 0:03.5]` 식으로 구간을 표시하고, 자막
  문구를 그대로 읽어도 되지만 말투는 좀 더 자연스럽게(예사말 → 높임말 등) 다듬어도 된다.
  마지막에 톤 가이드(예: "광고 멘트처럼 말하지 말고 친구한테 근황 얘기하듯")를 한 줄 남긴다.
- 사용자에게: "reel_silent.mp4를 CapCut 등에 불러와서 narration_script.txt 보면서 목소리
  녹음하시면 완성됩니다"라고 안내한다.

## 4. 캡션 + 해시태그 작성

- **톤**: `CLAUDE.md`/`SERVICE_DESIGN.md`에 드러나는 이 프로젝트의 목소리를 따른다 — 담백하고
  솔직한 "빌드 인 퍼블릭" 톤. 과장된 광고 문구보다 "이번에 이런 걸 만들었다 / 이런 문제를
  풀었다"는 식의 진행 로그에 가깝게 쓴다.
- **구조**:
  1. 첫 줄: 이번에 뭘 했는지 한 문장 훅
  2. 2~3줄: 왜 이걸 만들었는지 / 어떤 문제를 풀었는지 (필요하면 `SERVICE_DESIGN.md`의 동기·
     배경 참고)
  3. 마지막 줄: 다음에 뭘 할지, 또는 가벼운 구독 유도 (실제 URL을 캡션 본문에 박을 필요는 없음 —
     "프로필 링크 확인" 정도로 충분)
- **해시태그**: 한국어+영어 섞어서 10~15개. 카테고리별로 고른다:
  - 프로젝트 성격: #빌드인퍼블릭 #사이드프로젝트 #스타트업 #뉴스레터
  - 타겟 고객: #취준생 #직장인 #이직준비 #커리어
  - 기술/도구: 이번 작업이 개발 관련이면 #클로드코드 #AI개발 등 (마케팅/디자인 작업이면 생략)
  - 산업: 이번에 다룬 산업이 있으면 그것도 추가 (예: #로봇산업 #AI산업)
- `instagram_posts/<날짜>/caption.txt`에 저장한다. 형식: 캡션 본문 → 빈 줄 → 해시태그 한 줄
  나열. 인스타그램 앱에 그대로 복사-붙여넣기 하기 좋게 만든다.

## 5. 마무리

- 완료되면 사용자에게 저장된 이미지/영상 경로와 캡션 미리보기를 보여준다. 영상을 만들었으면
  녹음 대본(`narration_script.txt`) 사용법도 같이 안내한다.
- `instagram_posts/.last_post_commit`에 현재 `git rev-parse HEAD` 값을 기록해, 다음 실행 때
  "그 이후 변경사항"만 다루도록 한다.
- `instagram_posts/`는 `.gitignore`에 이미 등록돼 있다 — 마케팅 초안이라 저장소에는 커밋하지
  않는다. 새로 만들 때마다 이 규칙이 유지되는지 확인한다.
