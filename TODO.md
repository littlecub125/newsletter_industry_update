## 장기 구상: 에이전트 분리 아키텍처 (2026-07-12 사용자 제안, 나중에)

지금은 사람(운영자)이 파이프라인 스크립트를 손으로 실행하고 `tagged_articles.jsonl`/
`news.json`을 직접 검수·관리한다. 사용자가 제안한 다음 단계는 이 과정을 역할별로 나누는
것 — 아래는 원안 + 검토 후 다듬은 순서/제약이다 (2026-07-12, 원안 검토 반영).

**원안 4가지:**
1. 개발 에이전트가 관리자 페이지를 만든다 — 카드뉴스 내용(요약/`why_it_matters`/산업 분류
   등)을 사람이 편집.
2. 기사를 스크래핑해오는 에이전트.
3. 홈페이지 관리 에이전트.
4. 홈페이지 관리 에이전트가 관리자 페이지를 통해 기사를 업로드.

**검토 후 다듬은 순서 및 제약:**

1. **1번(관리자 페이지)부터 먼저, 2·3번은 그다음.** 실제 가치는 관리자 페이지(지금의 수동
   JSONL 편집을 없애주는 것)에 압도적으로 크고, 스크래핑/홈페이지 쪽은 사실 지금 있는
   스크립트(`scrape_rss.py`+`tag_articles.py`, `build_news.py`+배포)를 스케줄러에 올리는
   것뿐이라 나중에 붙여도 무방하다. `CLAUDE.md`의 "몇 호 수동 발행해보고 반복 노동인
   곳만 자동화" 원칙과도 맞는다.
2. **관리자 페이지는 정적 사이트로는 못 만든다.** 지금 웹사이트는 GitHub Pages(정적
   HTML)라 서버가 없다. 편집·저장이 되려면 최소한 백엔드(API + 데이터 저장소)와 **인증
   (로그인)**이 필요 — "web/ 폴더 추가"로 끝나는 작업이 아니다. 처음엔 인터넷에 올릴 필요
   없이 로컬에서만 도는 도구(로컬 서버 + 브라우저)로 시작해도 충분할 수 있다.
3. **"에이전트"의 정의를 먼저 정할 것.** AI가 판단해서 움직이는 자율 에이전트인지, 아니면
   기존 스크립트를 정해진 시간에 도는 예약 작업(스케줄러)인지에 따라 구현 난이도가 완전히
   다르다. 스크래핑·홈페이지 쪽은 이미 로직이 정해져 있어(RSS 수집 규칙, 배포 순서) AI
   판단이 딱히 필요 없고 단순 스케줄러로 충분해 보인다. "에이전트"라는 표현은 관리자
   페이지에서 사람이 검수하는 흐름에만 어울린다.
4. **지금까지 발견한 문제를 처음부터 설계에 넣을 것:**
   - 한영 중복/같은 사건 중복 게재 문제 — 자동화 전에 최소한의 중복 필터가 없으면 자동화가
     문제를 더 빨리 반복시킴.
   - 뉴스와이어 "하루 5건" 제한 — 지금은 사람이 눈으로 확인하는데, 자동화되면 이 체크포인트가
     사라지니 코드로 강제하는 안전장치가 필요.
   - API 비용 — 사람 확인 없이 스케줄러가 계속 돌면 예산 초과 위험. 회차당/일일 상한선 필요.
5. **홈페이지 에이전트는 "승인 상태"가 있는 데이터만 배포해야 함.** 관리자 페이지에 단순히
   "존재하는 항목"이 아니라 사람이 명시적으로 체크한 **"승인됨" 플래그**가 있어야 한다.
   안 그러면 사람이 미처 못 본 항목까지 실수로 배포될 위험이 있음.
   - **(2026-07-20 리뷰 추가) 이 플래그는 지금 설계대로면 기사 단위 단일 플래그다.**
     그런데 티어별로 노출 대상이 다르다(무료=공개 웹, VIP=회사 지정 발송 등) — 프리미엄/
     VIP를 실제로 열면 같은 기사도 "이 티어엔 승인, 저 티어엔 아직" 같은 상황이 생길 수
     있음. 지금(무료 티어만 운영)은 문제 없지만, 프리미엄/VIP 열 때 플래그를
     티어별/발송대상별로 세분화할지 다시 판단할 것.

**연결 흐름(다듬은 버전)**: 스크래핑 스케줄러(중복/예산 가드 포함)가 수집·태깅 →
관리자 페이지(우선 로컬 도구)에서 사람이 검수·수정·**승인 플래그 체크** → 홈페이지 자동
배포는 승인된 항목만 읽어서 반영.

**왜 지금 안 하는지**: `CLAUDE.md`의 "이 프로젝트의 우선순위" — 기술적 완성도보다 첫 호
발행과 구독자 확보가 먼저다. 자동화는 실제로 몇 호를 수동 발행해보고 어디가 진짜 반복
노동인지 확인한 뒤에 붙이기로 이미 정해져 있음. 지금 막 파이프라인 1차 테스트를 끝낸
단계라, 이 구상은 첫 호 발행 이후 실제 운영 부담이 확인되면 다시 꺼낼 것.
